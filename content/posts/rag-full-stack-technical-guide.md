在 Agent 的开发过程中，RAG（检索增强生成）的应用水平直接决定了 Agent 的业务上限。但在实际落地中，很多团队面临三个共性问题：**知识库构建缺乏标准、检索召回精度不够、缺少量化评测体系**。

本文从大淘宝技术团队分享的 RAG 全链路技术指南出发，系统梳理 RAG 从文档加载到评估的每一个核心环节，帮助你构建**可测、可调、可信赖**的 RAG 应用。

## 为什么需要 RAG？

大型语言模型虽然强大，但在处理特定领域知识、超出训练数据范围的查询或需要实时信息的场景时，仍然会产生"幻觉"。RAG 的核心思路很直接：

> 当用户提问时，系统先从外部知识库中检索相关信息，将这些信息作为参考背景与问题一起提供给大模型，让模型像做**开卷考试**一样，基于可靠证据生成更准确、可溯源的回答。

RAG 解决的问题不仅仅是准确性，还包括**知识时效性**、**私有数据安全**和**回答可追溯性**。

## RAG 全链路架构

一套完整的 RAG 系统包含以下核心环节：

```
文档加载 → 智能切分 → 索引构建 → 检索优化 → 生成调优 → 评估闭环
                                                      ↓
                                              Graph RAG（进阶）
```

下面逐一拆解。

## 一、文档加载：RAG 的入口

文档加载的质量直接决定了后续检索的准确性。它的核心任务是将各种格式的非结构化数据转化为系统可处理的统一文本格式。

### 四个关键步骤

1. **多格式适配** — 兼容 PDF、Word、Markdown、HTML、JSON、TXT、Excel 等
2. **内容提取** — 从原始文件中剥离净文本。PDF 需解析文字层，扫描件需 OCR；HTML 需剔除脚本和样式表
3. **元数据提取** — 记录文件名、页码、标题、作者、章节层级等属性
4. **初步清洗** — 去除乱码、多余空格、非打印字符等冗余信息

文档加载阶段看似简单，但做得好与不好，后续环节的差距会成倍放大。

## 二、智能切分：决定检索颗粒度

文档切分是 RAG 系统中最关键的技术决策之一。切分粒度不当，要么丢失上下文，要么引入过多噪声。

### Meta-Chunking：基于语义的切分

Meta-Chunking 是一种基于逻辑和语义的智能切分方法，核心原理是计算句子间的**PPL（困惑度）**来判断切分边界。

**PPL 的直观理解：**
- **低 PPL** → 语言通顺、逻辑连贯
- **高 PPL** → 内容突兀、逻辑断层

当模型处理到某一句时，如果 PPL 突然大幅升高，说明这一句与前面内容在逻辑上"脱节"了，这个点就是潜在的切分边界。

### 语义补全：解决切分后的上下文丢失

切分必然导致信息碎片化，Meta-Chunking 通过以下步骤进行语义补全：

1. **全局增强的文本块重写** — 用 embedding 模型识别与当前文本块相近的其他块，利用 LLM 判断是否丢失信息
2. **丢失信息提炼与补全** — 对检测到的缺失信息进行打分筛选，防止引入无关内容
3. **摘要生成** — 为每个文本块生成富含全局信息的简明摘要，增强语境感知能力

## 三、索引构建：Embedding 模型详解

索引构建的目标是将文档按语义映射到向量空间。现代 embedding 模型（如千问 text-embedding-v4、BGE 等）基于 Transformer 架构，将文本转换为稠密向量。

### Embedding 生成过程

以"苹果手机很好用"为例，映射过程如下：

1. **分词与编号（Tokenization）**
   - 切分：`[苹果, 手机, 很好用]`
   - 添加特殊 token：`[CLS, 苹果, 手机, 很, 好用, SEP]`
   - 转换为词表 ID

2. **映射初始向量**
   - 查表：每个 ID 映射为固定维度的初始向量（如 1536 维）
   - 通过三个权重矩阵（W_Q、W_K、W_V）生成 Q、K、V 三个角色

3. **注入位置信息** — 让模型知道每个 token 的顺序

4. **自注意力机制（Self-Attention）**
   - 计算每个 token 与其他 token 的相关性
   - "苹果"的 Q × "手机"的 K 会得到高分（语义强相关）
   - "苹果"的向量会吸收"手机"的特征，抑制"水果"特征，放大"电子产品"特征

5. **逐层深化** — 以上过程重复 12 层或 24 层，每个 token 的向量最终包含整句话信息的"超级词向量"

6. **池化处理（Pooling）**
   - **CLS**：取 [CLS] 位置的向量
   - **Mean Pooling**：所有向量取平均（BGE 等模型最常用）
   - **Max Pooling**：取每维度的最大值

7. **L2 归一化** — 归一化后，点积直接等于余弦相似度，大幅提升检索速度

### 常用 Embedding 模型

| 模型 | 维度 | 特点 |
|------|------|------|
| BGE-M3 | 1024 | 多语言、多粒度，支持稠密/稀疏检索 |
| 千问 text-embedding-v4 | 1024 | 中英文优化，长文本能力强 |
| text-embedding-3-large | 3072 | OpenAI 出品，综合能力强 |
| GTE-Qwen2-7B | 3584 | 开源大模型，检索精度高 |

## 四、检索优化：提升召回精度

检索阶段的目标是召回与问题最相关的文本段。这一步的质量直接决定了生成答案准确性的上限。

### 1. Query 改写

**核心目标**：纠正用户原始查询中的偏差、模糊或不完整信息。

解决的关键问题：
- **指代消解**：多轮对话中"它的价格是多少？" → "华为 Mate 60 Pro 的价格是多少？"
- **纠错与去噪**：修正错别字，剔除口语化冗余
- **术语对齐**："肚子疼怎么办" → "腹痛的治疗建议"

### 2. 多查询生成

将单一查询转化为 3-5 个不同角度的查询，通过"广撒网"提升召回率。

> 用户问"如何提高睡眠质量？"
> - 侧重生活习惯："睡前哪些习惯有助于入睡？"
> - 侧重环境因素："什么样的卧室环境适合深度睡眠？"
> - 侧重医学建议："改善失眠的科学方法有哪些？"

相当于在向量空间里投下多枚"探测器"，防止因提问角度片面而漏掉关键资料。

### 3. HyDE：用假设文档增强检索

**HyDE（Hypothetical Document Embeddings）** 的思路很巧妙：不是用"问题"去搜"答案"，而是先让 AI 编一个"假答案"，再用这个"假答案"去搜"真内容"。

**为什么有效？** 传统向量检索面临"非对称检索"问题——用户问题短且带疑问语气，而知识文档长且是陈述性的。HyDE 将"问题-文档匹配"转换成了"文档-文档匹配"，语义对齐度大幅提升。

### 4. 反向 HyDE（Doc2Query）

反向 HyDE 的优势在于**可离线处理**，不影响实时响应时间。

针对每个文本块（chunk），让 LLM 生成 3-5 个可能的"虚构问题"，然后针对这些问题建立索引，关联到原始文本。

> 原始文本："员工在非一线城市出差，每日住宿补贴上限 350 元，需在 15 个工作日内提交发票报销。"
> 
> 生成的虚构问题：
> - "出差去二线城市住酒店一天能报销多少钱？"
> - "出差回来后最晚什么时候报销？"
> - "报销差旅费需要什么类型的发票？"

当用户问"我想报销上周去西安出差的房费，该怎么办？"时，直接比对虚构问题的匹配度（可达 0.9）远高于比对原始文本（约 0.6）。

### 5. 标签过滤

将非结构化数据转化为半结构化数据，在语义向量检索的基础上引入**标签硬匹配**，解决"看着像但不是同一个东西"的问题。

> 搜索"iPhone 15 Pro Max"时，向量检索可能把它和"iPhone 14"放在一起（语义相似），但标签匹配能通过 `iphone15` 硬标签精准锚定目标。

### 6. 重排序（ReRank）

检索回来的文档仅说明与 Query 相关，但不能保证精确匹配。重排序是解决"**检索回来的资料看着挺像，但就是答非所问**"的最佳方案。

**原理**：利用交叉编码器（Cross-Encoder）将 Query 和候选文档拼在一起作为长句子输入模型。通过 Cross-Attention，Query 中的每一个 token 都可以直接观察到文档中的每一个词，捕捉极细微的匹配关系。

> "推荐一款 2000 元以下、续航好的华为手机"
> ReRank 会逐一检查：是不是华为？是不是低于 2000？有没有提到续航？缺一项，得分大幅跳水。

**优缺点权衡：**
- ✅ 精度飞跃、多源融合、降低大模型负担、缓解"中间丢失"现象
- ⚠️ 增加延时、有算力开销、对查询扩写不友好（多查询 × 5 倍计算量）

## 五、生成调优：让回答更可靠

检索到相关文本后，RAG 应用将问题与文本段通过 Prompt 模板输入大模型。这一阶段的核心是利用大模型的**总结能力**而非其**固有知识**。

### 常见问题

- 没有检索到相关信息时，大模型捏造答案
- 知识冲突：文档 A 说"可以办"，文档 B 说"禁止办"
- 丢失中间信息：LLM 对长上下文的开头和结尾敏感，中间内容容易被忽略
- 模型忽略参考资料，直接用训练时的陈旧知识回答

### 优化方式

1. **选择合适的大模型**
   - 简单信息查询 → 小参数模型即可
   - 复杂逻辑推理 → 大参数、强推理模型
   - 大量文档片段查阅 → 长上下文模型
   - 非通用领域（法律等） → 领域专用微调模型

2. **优化 Prompt**
   - 明确要求不编造答案，强制引用标注信息来源
   - 添加内容分隔标记，将 Prompt 和检索切片明确分开
   - 根据问题类型调整模板（框架型 vs 结论型）

3. **模型调参**
   - `temperature` / `top_p`：事实性内容调低，创造性内容调高
   - `presence_penalty`：适当调高可减少重复句式
   - `seed`：相同 seed 保证输出一致性
   - `max_tokens`：控制成本和响应时间

4. **SFT 微调**
   - 准备"问题-参考资料-标准答案"语料
   - 训练模型学会"根据参考资料回答"以及"资料不足时拒绝回答"

## 六、进阶：Graph RAG

Graph RAG 结合知识图谱和 LLM，解决传统 RAG 在处理复杂问题时的局限性。

### 解决的核心问题

**多跳问题**：传统 RAG 向量检索可能只找回包含"A 的导师"的片段，但不一定能找回"B 的导师"的片段。Graph RAG 通过图路径追踪，轻松找到答案。

**全局理解**：传统 RAG 擅长局部搜索，但不擅长总结全文。Graph RAG 通过社区检测算法，将图谱划分为不同层级的簇，预先生成摘要，实现全局概括。

### 工作原理

**第一阶段：索引构建**
1. 文本切分 → 2. 利用 LLM 提取实体与关系（三元组） → 3. 构建全局知识图谱 → 4. 社区检测（如 Leiden 算法）→ 5. 为每个社区生成总结

**第二阶段：查询检索**
- **局部检索（Local Search）**：定位实体 → 提取邻居节点信息 → 结构化数据喂给 LLM
- **全局检索（Global Search）**：检索预生成的社区摘要 → 汇总多个社区结论 → 给出全局回答

**代表性框架**：Microsoft GraphRAG、LlamaIndex、LightRAG

## 七、RAG 评估：Ragas 框架

Ragas（Retrieval Augmented Generation Assessment）使用 LLM-as-a-Judge 的理念，通过一系列自动化指标衡量 RAG 系统性能。

### 检索相关指标

| 指标 | 核心关注 | 输入 |
|------|----------|------|
| **Context Precision** | 相关文本块是否排在排序顶部 | user_input, reference, retrieved_contexts |
| **Context Recall** | 不遗漏重要结果 | user_input, reference, retrieved_contexts |

### 生成相关指标

| 指标 | 核心关注 | 输入 |
|------|----------|------|
| **Faithfulness** | 回答与检索内容的事实一致性 | user_input, response, retrieved_contexts |
| **Answer Relevancy** | 回答与用户问题的匹配度 | user_input, response |
| **Noise Sensitivity** | 提供错误回答的频率 | user_input, reference, response, retrieved_contexts |

其中**噪声敏感度**分为两类：
- **相关上下文噪声**：模型是否把相关文档中的冗余细节也搬进答案
- **不相关上下文噪声**：模型是否被不相干文档带偏

### 测试集自动生成

Ragas 提供了基于知识图谱的测试集生成工具，支持多种查询类型：

- **单跳查询（Single-Hop）**：从单个文档检索信息
- **多跳查询（Multi-Hop）**：需要跨越多个来源的推理
- **具体查询（Specific）**：基于事实的精确检索
- **抽象查询（Abstract）**：需要整合、归纳的宽泛回答

通过设定不同场景参数（节点、查询长度、查询风格、人设），可以自动生成覆盖真实交互情况的测试集。

## 总结

RAG 不是简单的"检索 + 生成"，而是一个需要精细调优的工程系统。从文档加载的入口质量，到切分策略的颗粒度选择，再到 embedding 模型选型、Query 改写、HyDE/Doc2Query 等检索增强手段、ReRank 精度跃升，以及最终的 Ragas 自动化评估——每一个环节都有大量的优化空间。

**核心认知**：

> 真正拉开 RAG 应用差距的，往往不是用了什么最新的模型，而是有没有在每个环节做到**可测、可调、可信赖**。

先把全链路的每一个环节搭起来、测起来、调起来，后面你会省掉很多重复劳动。
