每个团队都有自己的气质。有的团队以执行力著称，有的以创造力见长。我们这支团队，大概可以用"在代码里找到默契"来形容。

这篇文章是对我们团队的一次自我介绍——不是 HR 版本的，是工程师视角的。

## 我们是怎么走到一起的

团队的核心成员来自不同的背景：有在大厂历练过的，有从创业公司出来的，还有一直在做开源项目的独立开发者。大家因为对某些技术问题的相似判断聚在一起，慢慢形成了现在的团队。

我们没有刻意营造文化——文化是工作方式自然沉淀出来的。几年下来，有几件事变成了我们的默认行为：

- **遇到问题先写文档**：不急着开会讨论，先把问题和现有信息整理成文字，清晰表达往往比头脑风暴更有效
- **代码评审认真对待**：PR 不是走流程，是真正交流想法的地方，好的 Review 比好的代码本身更有价值
- **不崇拜复杂度**：把简单问题复杂化不是技术能力，反过来才是

## 团队构成

我们目前的团队分为几个方向：

### 前端工程

负责用户界面和交互体验。我们的前端工程师不只写页面，他们深度参与性能优化、可访问性、设计系统的建设。用他们的话说："前端是用户真正接触产品的地方，每一毫秒的延迟都是我们的责任。"

技术栈以 React + TypeScript 为主，辅以 Vite 构建工具链和 Storybook 组件管理。

### 后端工程

负责服务架构和 API 设计。我们的后端团队对系统设计有近乎偏执的追求，写 RFC（Request for Comments）是做任何重要改动之前的必经步骤——不是形式，是真的用来对齐认知。

主力语言是 Go，数据层以 PostgreSQL 为主，消息队列用 Kafka。

### 基础设施

这是个只有两个人的小组，但撑起了整个工程平台。Kubernetes 集群管理、CI/CD 流水线、可观测性体系——所有这些都出自他们之手。他们有一句名言："好的基础设施让人感觉不到它的存在。"

### 数据工程

负责数据仓库、ETL 流程和数据质量保障。数据团队是整个公司做决策的基础设施，他们最自豪的事是：业务团队看数据的时候从不需要来问"这个数字对不对"。

## 我们怎么工作

### 异步优先

我们团队分布在不同城市，同步时间有限。这倒逼我们养成了异步沟通的习惯：重要的事情写成文档，讨论发生在文档的评论区，会议只用来做"需要即时互动才能推进"的事。

异步工作的前提是**写作能力**。我们对团队成员的写作能力要求并不低于技术能力——能把问题说清楚，是工程师最核心的软技能之一。

### 工程师主导技术决策

我们没有专职的架构师。技术决策由最了解问题的工程师提出，通过 RFC 流程收集意见，最终由相关人达成共识。这个过程慢，但做出的决定质量更高，执行时阻力也更小。

### 度量自己，不是证明自己

我们不设置 KPI 式的代码行数、提交次数指标。我们关注的是：
- 系统可用性和性能是否在提升？
- 开发者体验（DX）是否在变好？
- 用户反馈中的工程类问题是否在减少？

这些指标更难造假，也更接近我们真正想要的结果。

## 我们在找什么样的人

如果你对我们感兴趣，以下几点比简历上的背景更重要：

1. **你能把复杂的事情讲清楚**——无论是写文档还是口头表达
2. **你对自己的工作有判断力**——不只会执行，也知道为什么这么做
3. **你乐于接受反馈**——好的 Code Review 文化需要双向的开放心态
4. **你对"够好了"保持警惕**——不是追求完美主义，而是不轻易满足

## 写在最后
