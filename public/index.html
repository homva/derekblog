Shannon 入门教程：用 AI 做自动化白盒渗透测试

2026-06-16 安全 #安全测试 #AI #渗透测试

前言：你的渗透测试还在一年一次吗？

大多数团队的安全现状是这样的：代码每天都在发，但渗透测试一年才做一次。中间那 364 天，漏洞可能已经在生产环境里裸奔了。

传统 SAST 工具（静态分析）能扫出一些潜在问题，但误报率高得离谱。开发看到一堆"可能的 SQL 注入"告警，最后发现大部分都是假阳性，久而久之就没人管了。

Shannon 的思路完全不同：它不只是"猜测"漏洞，而是真正去尝试利用。只有 exploit 成功的漏洞才会出现在最终报告里 —— 零误报。

这篇文章带你从零开始，完整跑通 Shannon 的安装、配置和第一次渗透测试。

什么是 Shannon？

Shannon 是由 Keygraph 开发的开源 AI 渗透测试工具，采用 AGPL-3.0 协议。它是一个 CLI 工具，通过 npx 分发，利用 Docker Worker 容器执行真实的渗透测试。

核心工作流程

1

源码分析 → 攻击面识别 → 漏洞分析 → 真实 Exploit → 生成报告

它不是一个被动扫描器，而是一个自主执行的 AI Agent：

  1. Pre-Reconnaissance：读取目标源码，识别框架、入口点、数据流
  2. Reconnaissance：探索运行中的应用，将运行时行为与代码上下文关联
  3. Vulnerability Analysis：针对不同漏洞类型（注入、XSS、SSRF 等）启动专项 Agent
  4. Exploitation：尝试真实的 PoC 攻击，无法证明的假设直接丢弃
  5. Reporting：将验证过的发现、证据和修复建议编译成 Markdown 报告

Shannon Lite vs Pro

维度Shannon Lite（开源）Shannon Pro（商业）
协议AGPL-3.0商业授权
测试模式仅白盒（需源码）白盒 + 黑盒
形态本地 CLISaaS 平台
覆盖范围OWASP 重点漏洞SAST、SCA、Secrets、IaC、容器等全生命周期
CI/CD手动/本地运行企业级 CI/CD 门禁
修复验证需手动重跑自动验证修复

Lite 版本对于本地项目级白盒测试完全够用，Pro 适合需要持续安全覆盖的企业团队。

RAG 全链路技术详解：从文档加载到评估的完整工程实践

2026-05-25 技术 #RAG #LLM #Embedding

在 Agent 的开发过程中，RAG（检索增强生成）的应用水平直接决定了 Agent 的业务上限。但在实际落地中，很多团队面临三个共性问题：知识库构建缺乏标准、检索召回精度不够、缺少量化评测体系。

本文从大淘宝技术团队分享的 RAG 全链路技术指南出发，系统梳理 RAG 从文档加载到评估的每一个核心环节，帮助你构建可测、可调、可信赖的 RAG 应用。

为什么需要 RAG？

大型语言模型虽然强大，但在处理特定领域知识、超出训练数据范围的查询或需要实时信息的场景时，仍然会产生"幻觉"。RAG 的核心思路很直接：

当用户提问时，系统先从外部知识库中检索相关信息，将这些信息作为参考背景与问题一起提供给大模型，让模型像做开卷考试一样，基于可靠证据生成更准确、可溯源的回答。

RAG 解决的问题不仅仅是准确性，还包括知识时效性、私有数据安全和回答可追溯性。

RAG 全链路架构

一套完整的 RAG 系统包含以下核心环节：

2
3

文档加载 → 智能切分 → 索引构建 → 检索优化 → 生成调优 → 评估闭环
                                                      ↓
                                              Graph RAG（进阶）

下面逐一拆解。

一、文档加载：RAG 的入口

文档加载的质量直接决定了后续检索的准确性。它的核心任务是将各种格式的非结构化数据转化为系统可处理的统一文本格式。

四个关键步骤

  1. 多格式适配 — 兼容 PDF、Word、Markdown、HTML、JSON、TXT、Excel 等
  2. 内容提取 — 从原始文件中剥离净文本。PDF 需解析文字层，扫描件需 OCR；HTML 需剔除脚本和样式表
  3. 元数据提取 — 记录文件名、页码、标题、作者、章节层级等属性
  4. 初步清洗 — 去除乱码、多余空格、非打印字符等冗余信息

文档加载阶段看似简单，但做得好与不好，后续环节的差距会成倍放大。

二、智能切分：决定检索颗粒度

文档切分是 RAG 系统中最关键的技术决策之一。切分粒度不当，要么丢失上下文，要么引入过多噪声。

问题原话后果
擅自假设“make wrong assumptions on your behalf and just run along with them”方向跑偏，返工

Karpathy 原文解读：「用 Claude 编码几周后的随机笔记」

2026-04-21 技术 #AI编程 #Claude #Karpathy

发布时间：2026年1月26日，760万浏览。

1️⃣ 编码工作流的范式转移

从 11 月的「80% 手写 + 20% Agent」到 12 月的「80% Agent + 20% 审查修改」

Karpathy 说自己现在主要在用英语编程——用自然语言告诉 LLM 写什么代码。他承认这有点伤自尊，但用"代码级动作"操作软件的效率太高了。

关键判断：这是他 20 年编程生涯中最大的工作流变化，而且发生在几周内。他认为已有「百分之十几」的工程师经历了类似转变，但大众认知还停留在个位数百分比。

📌 解读：这不是一个渐进过程，而是一个相变（phase shift）——能力一旦跨过某个阈值，工作流会在极短时间内翻转。

2️⃣ IDE 还在，Agent 群还没到时候

Karpathy 对两个 hype 泼了冷水：

  • ❌ “不再需要 IDE”——太早了
  • ❌ “Agent Swarm（多 Agent 群）"——也太早了

他现在的实际工作流：左边开几个 Claude Code 终端窗口，右边开 IDE 盯着代码做人工审查。像盯鹰一样盯着 AI。

错误类型变了：不再是语法错误，而是概念性错误——像一个粗心急躁的初级程序员会犯的那种。

🔴 他列出的具体缺陷（也是上一份调研中四大原则的来源）：

缺陷原文
擅自假设“make wrong assumptions on your behalf and just run along with them without checking”
不管理困惑“don’t manage their confusion, don’t seek clarifications”
不暴露矛盾“don’t surface inconsistencies”
不展示权衡“don’t present tradeoffs”
不会反驳“don’t push back when they should”
太迎合“still a little too sycophantic”
过度设计“overcomplicate code and APIs, bloat abstractions, don’t clean up dead code”
千行变百行“implement a bloated construction over 1000 lines… you say ‘couldn’t you just do this instead?’ → immediately cut it down to 100 lines”
越界修改“change/remove comments and code they don’t sufficiently understand as side effects, even if orthogonal to the task”

注意：他说即使写了 CLAUDE.md 做约束，这些问题仍然会发生。
